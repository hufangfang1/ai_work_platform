# 需求拆解:每项目需求子文档 + 共享接口契约

> 日期:2026-07-08
> 关联模块:AI 开发工单(BreakdownService / TaskService / PlanService / RunService)

## 1. 背景与问题

一份需求文档可能跨多个项目(例:`soe-job-select` 前端 + `soe-job-proxy` 后端)。当前拆解只产出:

- **一份共享 `breakdown_markdown`**(需求理解 / 涉及项目与分工 / 跨项目接口约定 / 风险点),站在需求负责人视角;
- 每项目一段 `scope_summary`(一句话分工)+ `interfaces`。

确认后每项目生成一张工单,但工单存的 `doc_content` 是**整篇原始需求文档**。更糟的是,原文被重复灌进下游三步:

| 步骤 | 代码位置 | 现在吃什么 |
|---|---|---|
| 生成开发文档(计划) | `PlanService::buildPrompt` (PlanService.php:106) | 整篇原文 + scope_summary + 读本项目代码 |
| 编码 | `RunService::buildPrompt` (RunService.php:251) | 整篇原文 + scope_summary + 已确认计划 |
| Review / fix | `RunService::buildPrompt` | 整篇原文 + 计划 + 反馈 |

**痛点**:前端工单在计划、编码、Review 全程都背着后端的 SQL 表、日志字段等噪音;拆解产出的是"总览 + 一句话分工",颗粒度和视角对不上"每个项目一份自己的落地需求"。

## 2. 目标

把跨项目原文,按每个项目在需求中的**职责**拆成**每项目一份落地需求子文档**(前端=页面/交互/字段展示;后端=接口/数据口径/表/日志),跨项目的接口约定收敛到**一份共享契约**作为唯一事实源。子文档成为下游计划/编码/Review 的干净输入,替代整篇原文。

## 3. 关键设计决策(已确认)

1. **每项目子文档 + 共享接口契约**(而非完全拆成 N 份独立文档)。跨项目接口约定只在共享契约里写一次,子文档引用它、不复述。
2. **两步生成**:第一步 AI 只做"共享需求理解 + 跨项目接口契约 + 每项目一句话分工";人工确认后,第二步按项目**各跑一次**生成该项目子文档,可单项目重跑。
3. **需求子文档 与 开发文档(计划)是叠加关系,不是二选一**:
   - 需求子文档(`spec_markdown`):讲**这个项目要做什么**,产品/接口视角,不读代码、不写文件路径。拆解阶段产出,供人 review。
   - 开发文档 / 计划(`plan_content`,已有):讲**在这个代码库里怎么做**,读真实代码、列文件/步骤/SQL。工单上产出。子文档喂给计划。

## 4. 单项目 / 多项目分流

涉及项目数 N 在拆解结束即可知(手动选 或 AI 判断产出的 `projects[]` 长度)。

- **N = 1(不拆分)**:不生成共享契约,不生成子文档,工单 `spec_markdown` 留空 → 下游直接沿用整篇原文(等于今天的行为,零回归)。一份需求 = 一张工单。
- **N > 1(才拆分)**:拆解产出 共享需求理解 + 跨项目接口契约 + 每项目一句话分工;逐项目生成 `spec_markdown`;下游读 子文档 + 契约。

## 5. 拆分依据:项目简介 → 职责

现有 prompt 已把项目 `description`(简介)喂进去(BreakdownService.php:183),但未明确要求以简介判定职责。改动:prompt 点明 **"依据每个项目的简介,判定该项目在本需求中的职责边界,只把属于它的部分划给它"**。前端简介写 H5/PC 展示 → 页面展示归它;后端简介写代理鉴权/日志 → 接口数据口径归它。

## 6. 完整数据流

```
原始需求文档(跨项目,整篇)
   │
   ├─ 第一步 分工拆解(改造现有 breakdown)
   │     产出:① 共享需求理解 + 跨项目接口契约(仅 N>1) + 风险点
   │            ② projects[]:每项目 { role(前端/后端), 一句话分工 }
   │     人工确认 → 每项目生成一张工单(现有逻辑不变)
   │
   ├─ 第二步 逐项目生成子文档(新增,仅 N>1,每张工单各跑一次)
   │     输入:原文 + 共享接口契约 + 本项目分工 + 项目简介/repo
   │     产出:task.spec_markdown(本项目视角落地需求)
   │            · 前端 → 页面结构/交互流程/展示字段与来源标注/脱敏/空异常态,不写 SQL
   │            · 后端 → 接口清单(入参出参口径)/数据来源分层/SQL 聚合口径/日志快照字段,不写页面
   │            · 跨项目交互一律"见共享接口契约第 X 节",不复述
   │
   └─ 下游(计划 / 编码 / Review)
         本项目上下文 = task.spec_markdown 有值 ? (子文档 + 共享契约) : 原文
```

## 7. 数据模型改动(最小)

- `ai_dev_breakdowns`:结构不动。约定 `content` 里"跨项目接口契约"是**独立且完整的一节**(共享契约载体);`projects_json` 每条为 `{project_id, project_name, role, scope_summary}`,**不塞子文档**。
- `ai_dev_tasks`:**新增列 `spec_markdown`**(本项目子文档,可空)。工单本就是 (需求 × 项目) 唯一一条,子文档挂工单上最自然。历史版本暂不做(YAGNI),重跑覆盖。

## 8. 触发时机与交互

- 拆解确认后:若 N>1,自动为每张工单排一个"子文档生成" run(异步)。
- 工单详情页:新增"重新生成本项目需求文档"按钮,支持单项目重跑。
- "生成开发文档(计划)"功能**不改交互、不挪位置**,仅把输入从原文换成 子文档+契约(有子文档时)。

## 9. 改动点清单

1. `BreakdownService::buildPrompt`:第一步 prompt 强化——只做理解+分工+契约,禁止展开单项目实现细节;明确"依据项目简介判定职责";契约独立成完整一节。
2. 新增 第二步子文档生成:service + run 类型(如 `task_spec`)+ 按 role 的前端/后端 prompt 模板;`finishRun` 落 `task.spec_markdown`。
3. `BreakdownService::confirm`:N>1 时,建工单后自动排子文档生成 run。
4. `ai_dev_tasks` 迁移:加 `spec_markdown` 列。
5. `PlanService::buildPrompt`、`RunService::buildPrompt`:注入上下文改为 `spec_markdown ?: doc.content`,并在有子文档时附带共享契约。
6. 前端工单详情页:展示 `spec_markdown` + "重新生成本项目需求文档"按钮。

## 10. 非目标(YAGNI)

- 不做子文档历史版本表(重跑覆盖)。
- 不做完全拆成 N 份独立文档(共享契约不下沉)。
- 单项目不生成"本项目视角整理稿",直接沿用原文。
